I have a MacBook Pro M4 with 48GB unified memory sitting on my desk. It can run a 27B parameter model at 12 tokens/second — faster than most cloud inference APIs. But I also have an AI assistant running 24/7 on a headless cloud VPS. The VPS has no GPU. It can't run local models.

What if I could combine them? Use the Mac's raw inference power from the VPS, without configuring VPNs, opening firewall ports, or paying for relay servers?

That's exactly what LM Link does. It's LM Studio's new P2P networking feature — and it took me under 5 minutes to set up.

LM Link creates an end-to-end encrypted mesh between your devices. No ports, no public IPs needed.

The Problem: GPUs at Home, Agents in the Cloud

If you're running AI agents, coding assistants, or automated workflows, you've probably hit this problem: your cloud server doesn't have a GPU, and your local machine with a GPU isn't always accessible.

The traditional solutions are painful:

  • Tailscale / WireGuard: Requires installation on every device, manual key management, subnet configuration, and network troubleshooting when NAT traversal fails.
  • Cloud GPU instances: An A10 (24GB VRAM) on Lambda Labs costs $0.86/hr — that's $627/month if you leave it running 24/7. An A100 is $1.48/hr ($1,080/month). Not practical for background inference.
  • ngrok / Cloudflare Tunnel: Exposes a local port to the internet. Works, but you're routing through a third-party relay, adding latency and creating a public endpoint you need to secure.
  • SSH tunnels: Fragile. They break when your connection drops. They don't auto-reconnect. You'll spend more time debugging tunnels than building things.

LM Link solves this differently — it's a P2P mesh VPN built directly into LM Studio, powered by Tailscale under the hood, but with zero configuration required from you.

What is LM Link?

LM Link is a feature in LM Studio that lets you connect multiple devices together into a private, encrypted network. Once connected, any model loaded on any device becomes available on every device in the network — as if it were running locally on localhost:1234.

Key details:

  • End-to-end encrypted: Built on Tailscale mesh VPNs. Your data never touches LM Studio's servers — only device metadata is shared for discovery.
  • No port forwarding: Tailscale handles NAT traversal automatically. Works behind routers, firewalls, and carrier-grade NAT.
  • OpenAI-compatible API: Once linked, your local localhost:1234 endpoint transparently proxies to remote models. Any tool that works with LM Studio (Codex, Claude Code, OpenCode, custom scripts) works with LM Link without changes.
  • Free tier: Up to 2 users, 5 devices each (10 devices total) — no credit card required.
  • Headless support: Works on servers without a GUI via llmster, LM Studio's headless daemon.

Setup: 4 Commands, 5 Minutes

The entire setup is 4 commands. No configuration files, no network settings.

Here's exactly what I did to connect my MacBook Pro to my Hetzner cloud VPS.

Step 1: Install LM Studio (Mac) + llmster (VPS)

On your Mac, download LM Studio from lmstudio.ai and load your model. I'm running Qwen 3.6 27B (Q4_K_M quantization, ~17GB) — a dense 27B parameter model from Alibaba that outperforms much larger models on coding benchmarks.

On your headless server, install llmster — LM Studio's CLI-only daemon:

# Install llmster on any Linux server
curl -fsSL https://lmstudio.ai/install.sh | bash

# Add to PATH
export PATH="$HOME/.lmstudio/bin:$PATH"

⚠️ On Ubuntu, ldconfig must be on your PATH. If the install fails, run: export PATH="/usr/sbin:$PATH" before the install command.

Step 2: Login on Both Devices

# On the VPS — shows a pairing code
lms login

# Visit https://lmstudio.ai/pairing in your browser
# Enter the code shown in the terminal
# Authentication successful!

On your Mac, if you're using the LM Studio GUI, you can log in from the app settings. On headless servers, the CLI gives you a pairing code to enter in your browser. No API keys to manage — just a one-time browser auth.

Step 3: Enable LM Link

# On the VPS
lms link enable
# LM Link is enabled and online.

# Check status
lms link status
# This device: ubuntu-8gb-hel1-2
# Status: Online
#
# Found 1 device:
#   - Mac.lan
#     Status: connected
#     Loaded Models:
#       - qwen/qwen3.6-27b

That's it. The devices discovered each other automatically. No IP addresses, no port numbers, no configuration files.

Step 4: Start the Local Server

# Start the OpenAI-compatible server on the VPS
lms server start -p 1234

# Test it
curl http://localhost:1234/v1/models
# Returns ALL models across your linked devices!

The server on the VPS now acts as a transparent proxy. When you call localhost:1234, it routes to whichever device has the model loaded — over the encrypted mesh.

Vision Support Works Too

LM Link transparently passes through all API features, including multimodal vision requests. I tested it with a base64-encoded image:

curl http://localhost:1234/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{
    "model": "qwen/qwen3.6-27b",
    "messages": [{
      "role": "user",
      "content": [
        {"type": "text", "text": "What is this image?"},
        {"type": "image_url", "image_url": {
          "url": "data:image/png;base64,..."
        }}
      ]
    }],
    "max_tokens": 100
  }'

The response came back correctly identifying the image content — the Android mascot in this case — processed entirely on the MacBook's GPU, routed through the encrypted mesh to my VPS. Vision inference adds some latency but is entirely usable for document analysis, screenshot understanding, and diagram interpretation.

⚠️ One limitation: LM Link's vision endpoint currently only accepts base64-encoded images, not URLs. You'll need to download and encode images client-side before sending them.

Practical Use Cases

Why would you want this setup? Here's how I'm using it:

  • AI agents on headless servers: My AI assistant runs 24/7 on a VPS. With LM Link, it can use powerful local models for tasks that don't need cloud-grade intelligence — saving API costs dramatically.
  • Coding assistant backends: Tools like Codex, Claude Code, and OpenCode can point to localhost:1234 and use your linked models. No changes to the tool configuration needed.
  • Multi-GPU pooling: Have a GPU rig at home and a Mac at the office? Link them both. Load different models on each device, and route requests to whichever has the right model.
  • CI/CD inference: Run llmster in your CI pipeline and use LM Link to access models without deploying API keys or paying per-token costs.
  • Embeddings: LM Link also exposes embedding models. I have text-embedding-nomic-embed-text-v1.5 available alongside Qwen — useful for RAG pipelines running on remote servers.

Disabling Thinking Mode

Qwen 3.6 has a "thinking" mode enabled by default — it spends tokens on an internal reasoning chain before producing output. This is great for complex reasoning but wasteful for simple tasks. With thinking mode on, a 5-word response consumed 500+ tokens.

To disable it, go to your model settings in LM Studio and turn off reasoning. The difference is dramatic:

  • Thinking ON: 9 tokens of actual output + 491 tokens of reasoning = 500 total tokens
  • Thinking OFF: 9 tokens of actual output + 0 reasoning = 9 total tokens

That's a 55x reduction in token usage for simple tasks. Always turn this off when you're using the model for straightforward generation.

What I'd Improve

LM Link is still in Preview, and a few things could be better:

  • No headless login: The lms login command requires entering a pairing code in a browser. There's no API-key or device-token auth for fully automated server setup. You need manual interaction once.
  • No systemd integration: llmster doesn't ship with a systemd service file. If the daemon crashes, it won't auto-restart unless you set that up yourself.
  • Vision requires base64 only: URL-based image passing doesn't work through the proxy. Every image needs to be downloaded and encoded client-side, which adds complexity and memory overhead.
  • No load balancing: If the same model is loaded on multiple devices, there's no way to distribute requests across them. You'd need to build that layer yourself.
  • No rate limiting or access control: Anyone with access to localhost:1234 on your server can use your linked models. Secure it with firewall rules or a reverse proxy.

If you have a Mac with decent unified memory (or any machine with a GPU) and you're running AI workloads in the cloud, LM Link lets you get free inference without sacrificing the convenience of cloud deployment.
